Role Definitions
admin
Full read/write across all resources. Can delete tickets, manage saved replies, configure SLA/escalation rules, view all staff data.
billing_specialist / support_staff
Read/write tickets and conversations assigned to them or unassigned. Cannot delete. Cannot modify SLA rules.
clinician
Can submit tickets and send chat messages. Can read their own tickets and conversations only. Cannot see internal notes.

?? SLA Breach - Overdue
Trigger: Ticket SLA deadline has passed (elapsedHours = overdueAfterHours for priority).
Scope: All open tickets. Runs every 2 minutes in browser; also via pg_cron job server-side.
De-dupe: sessionStorage key sc_notif_overdue_{ticketId} prevents repeat notifications per session.
Browser Push In-App Bell Email (server) DB notifications table
?? SLA At Risk
Trigger: Ticket has elapsed = 75% of its SLA window (atRiskPct = 0.75).
Scope: Urgent tickets within 18 h; High Priority within 54 h; Routine within 126 h.
De-dupe: sessionStorage key sc_notif_atrisk_{ticketId} per session.
Browser Push In-App Bell

Server-Side Notification Edge Function (pseudocode)
// supabase/functions/send-notification/index.ts
// Invoked by pg_cron (every 15 min) or DB trigger via HTTP

export default async function handler(req) {
  // 1. Load all Open tickets with SLA rules
  const tickets = await supabase.from('support_tickets')
    .select('*, sla_rules!priority(*)')
    .not('status', 'in', ['Completed','Closed','Archived']);

  for (const t of tickets.data) {
    const rule = t.sla_rules;
    const elapsedH = (Date.now() - new Date(t.date_submitted).getTime()) / 3600000;
    const pct = elapsedH / rule.overdue_after_hours;

    if (pct >= 1 && !t.sla_breached_at) {
      // Mark breach and notify
      await supabase.from('support_tickets')
        .update({ sla_breached_at: new Date().toISOString() })
        .eq('id', t.id);
      await notifyAdmins({
        type: 'sla_breach',
        title: `SLA Overdue: ${t.id}`,
        body: `Ticket ${t.id} (${t.priority}) has passed its SLA deadline.`,
        entity_type: 'ticket', entity_id: t.id
      });
    } else if (pct >= rule.at_risk_pct && !t.sla_at_risk_notified_at) {
      await supabase.from('support_tickets')
        .update({ sla_at_risk_notified_at: new Date().toISOString() })
        .eq('id', t.id);
      await notifyAssigned(t.assigned_user_id, {
        type: 'sla_breach',
        title: `SLA At Risk: ${t.id}`,
        body: `Ticket ${t.id} is approaching the SLA deadline.`,
      });
    }
  }
}

async function notifyAdmins(payload) {
  // Fetch all admin user_ids, insert a notifications row per user
  const admins = await supabase
    .from('profiles')
    .select('id')
    .in('role', ['admin', 'billing_specialist']);
  const rows = admins.data.map(u => ({ user_id: u.id, ...payload }));
  await supabase.from('notifications').insert(rows);
  // Optionally send email via Resend / Postmark
  if (payload.type === 'sla_breach') await sendEmail(admins.data, payload);
}
Standard Escalation Flow
1
Ticket Created
Staff or clinician submits a support ticket. Default status = New, workflow_stage = New Intake. SLA clock starts at date_submitted.
2
Triage & Assignment
Admin or billing specialist picks up the ticket. Stage ? Assigned. Assigned staff notified in-app and by email.
3
SLA At-Risk Warning (75%)
System detects ≥ 75% of SLA elapsed. Browser push + in-app notification sent to assigned staff. sla_at_risk_notified_at set.
4
SLA Breach (100%)
Ticket progresses to Overdue SLA status. Notification sent to all admins. sla_breached_at stamped in DB. Ticket appears in Command Center.
5
Auto-Escalation Rule Match
checkEscalationTriggers() fires. Matching rules produce escalation_item events. Admins notified. Ticket visible in Escalations tab.
6
Manual Escalation
Staff clicks "Escalate" in chat or sets workflow_stage = Escalation. Status ? Escalated. Triggers CRITICAL notification to all admins.
7
Convert to Billing Alert
For financial escalations, staff clicks "?? Billing Alert" in ticket drawer. Linked billing_alert created. Ticket history updated. Admin + billing specialist notified.
8
Resolution & Closure
Staff selects resolution code, clicks Close Ticket. Status ? Closed. workflow_stage ? Archived. closed_at stamped. Clinician notified.
Chat ? Ticket Escalation Flow
1
Clinician Opens Chat Widget
chat-widget.js creates a conversation record. Clinician presence tracked via Supabase channel.
2
Staff Identifies Ticket-Worthy Issue
Staff marks conversation as Urgent or identifies a billing/coding issue requiring formal tracking.
3
Click "??? Ticket" Button
convertToTicket() in admin-chat.html creates a support_ticket with the chat transcript. Conversation status ? Escalated. linked_ticket_id set.
4
Ticket Managed in Support Center
Full ticket workflow applies. Chat conversation remains open for follow-up messages.

How SLA is Computed
Elapsed hours = (now - ticket.date_submitted) / 3,600,000
SLA % = elapsed_hours / overdue_after_hours
Under SLA = SLA% < 75%
At Risk = 75% = SLA% < 100%
Overdue = SLA% = 100%

The clock stops when status changes to Completed, Closed, or Archived.
Tickets in "Waiting on Client" or "Waiting on Insurance" status do not pause the SLA clock - follow up promptly.
