HIPAA & Security Design Principles
All tables containing PHI inherit row-level security (RLS) scoped to organization_id. login_sessions stores only a hashed token — never the raw session token. audit_logs is append-only: no UPDATE or DELETE permissions should be granted on this table. Sensitive columns (SSN, DEA#, payment tokens) should be encrypted at rest via Supabase Vault or equivalent. All foreign keys reference uuid primary keys — never expose sequential integer IDs in URLs.

Entity Relationships — Part 1
How the 11 tables connect. All relationships are enforced by foreign key constraints.
From Table Cardinality To Table Via Column ON DELETE Notes
organizations 1:1 subscriptions subscriptions.organization_id CASCADE One active subscription per org
organizations 1:N locations locations.organization_id CASCADE Org → many service sites
organizations 1:N users users.organization_id SET NULL Users can exist without an org (super admins)
organizations 1:N providers providers.organization_id RESTRICT Cannot delete an org that has active providers
users 1:1 providers providers.user_id CASCADE Every provider is a user; not every user is a provider
users 1:N user_roles user_roles.user_id CASCADE User can have multiple roles across orgs
users 1:N login_sessions login_sessions.user_id CASCADE Multiple concurrent sessions supported
users 1:N audit_logs audit_logs.user_id SET NULL Audit rows kept even if user deleted
providers 1:N provider_specialties provider_specialties.provider_id CASCADE Provider can have multiple specialty tags
providers M:N providers clinician_supervisors.supervisee_id / supervisor_id RESTRICT Self-referential; junction table is clinician_supervisors
locations 1:N user_roles user_roles.location_id SET NULL Role can be scoped to one location; NULL = all locations
login_sessions 1:N audit_logs audit_logs.session_id SET NULL Track which session performed each action
